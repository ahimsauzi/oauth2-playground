# Post-Redirect-Get: 307 vs 303

**Discovered:** 2026-05-03 while building the guided PKCE flow

## What broke

The guided step 4 was calling the token endpoint twice on every button click. First call consumed the authorization code (200). Second call failed because the code was already gone (400). The user saw "Invalid or expired code" every time.

## Root cause

FastAPI's `RedirectResponse` defaults to HTTP 307 (Temporary Redirect). RFC 7231 says 307 **preserves the original HTTP method**. So a POST that receives a 307 redirect causes the browser to POST again to the new URL. This is the opposite of what you want after a form submission.

In the guided flow:
1. Browser POSTs to `/guided/pkce/step4`
2. Handler calls auth server `/token` (success, code consumed)
3. Handler returns `RedirectResponse(url="/guided/pkce/step4")` with default 307
4. Browser re-POSTs to `/guided/pkce/step4` (preserving method per 307 spec)
5. Handler calls `/token` again (fails, code already gone)

## Fix

Use HTTP 303 (See Other) for all redirects after form POST handlers. 303 explicitly instructs the browser to GET the redirect target regardless of the original method. This is the Post-Redirect-Get (PRG) pattern.

```python
# Wrong - browser re-POSTs
return RedirectResponse(url="/next-step")

# Correct - browser GETs the redirect target
return RedirectResponse(url="/next-step", status_code=303)
```

## Why this matters in real OAuth

Authorization codes are single-use by spec (RFC 6749). Any implementation that calls the token endpoint twice for the same code is either buggy or under attack. Real authorization servers detect code reuse and may revoke all tokens issued to that client (replay attack mitigation).

307 vs 303 is a common source of double-submission bugs in any web app that uses POST forms with redirects.
